# FedJD 实验研发总览

## 目的

这套文档用于控制 `FedJD` 从实验性玩具验证到完整 benchmark 的研发周期。当前联邦训练框架视为成熟可用，本项目只替换并扩展其中的多目标优化核心：将传统梯度型更新替换为基于 Jacobian 的服务器端全局方向决策。

本目录的目标不是一次性给出完美方案，而是提供一套足够稳定、可执行、可迭代修订的实验控制框架，使后续与其他 AI 或人工协作时不至于反复推倒重来。

## 文档结构

- `00_unified_protocol.md`
  统一实验规范。定义实验命名、日志、随机种子、对照、资源记录、停止准则。
- `01_stage_baseline_smoke.md`
  阶段 1：先跑通最小可用 FedJD 基线。上传完整局部 Jacobian，验证方法链路、数值稳定性与正确性。
- `02_stage_comm_profile.md`
  阶段 2：系统考察完整上传 Jacobian 的通信与时间代价，得到 FedJD 的代价画像。
- `03_stage_comm_reduction.md`
  阶段 3：研究通信压缩与加速方案，目标是在尽量不损失优化表现的前提下降低通信成本。
- `04_stage_full_benchmark.md`
  阶段 4：进入正式 benchmark，对比现有方法，在多类数据集和联邦场景下做完整测试。
- `05_stage_release_validation.md`
  阶段 5：形成论文级与开源级证据链，做复现实验、消融、鲁棒性和最终封版检查。

## 研发主线

### 阶段 0：统一协议先于实验堆叠
先固定：
- 什么是一次有效实验
- 什么是一次有效对比
- 什么日志必须保存
- 什么结果能作为阶段结束条件

### 阶段 1：只回答“FedJD 能不能正确跑通”
先不追求大模型、也不追求最好结果。先确认：
- 服务器端 Jacobian 聚合无误
- 局部 Jacobian 计算维度无误
- 更新方向可执行且数值稳定
- 在玩具问题上有合理趋势

### 阶段 2：量化完整 Jacobian 上传的代价
回答：
- 时间慢在哪里
- 内存占在哪里
- 上传字节数如何随 `m`、`d`、参与客户端数变化

### 阶段 3：通信压缩与工程折中
在阶段 2 已经定位瓶颈后，再引入压缩与近似，否则容易盲目优化错误对象。

### 阶段 4：正式 benchmark
回答研究问题：
- FedJD 是否优于或至少可比于 FMGDA/FSMGDA 等现有方法
- 在何种场景下优势明显
- 代价是否可接受

### 阶段 5：论文级封版
回答：
- 结果是否稳健
- 是否可复现
- 是否有足够消融支持结论

## 每阶段通用交付物

每个阶段结束后至少产出：
- 一份阶段总结 markdown
- 一份实验运行清单
- 一份结果表格或可视化图
- 一份“下一阶段是否放行”的结论

## 当前建议顺序

1. 先按 `01_stage_baseline_smoke.md` 做最小基线。
2. 确认代码逻辑与指标无误后，再进入 `02_stage_comm_profile.md`。
3. 只有在代价瓶颈被量化后，才进入 `03_stage_comm_reduction.md`。
4. 只有形成至少一个表现稳定的压缩版后，才进入 `04_stage_full_benchmark.md`。
5. 最后用 `05_stage_release_validation.md` 做最终封版。
