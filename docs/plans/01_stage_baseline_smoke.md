# 阶段 1：FedJD 基线跑通与玩具验证

## 1. 阶段目标

本阶段只回答一个问题：

**在完整上传局部 Jacobian 的设置下，FedJD 能否作为一个 server-centric 联邦多目标优化方法稳定跑通。**

这里不追求大规模，不追求最优性能，不追求通信效率。只要求：
- 算法链路正确
- 维度正确
- 指标能采集
- 玩具问题上有合理优化趋势

## 2. 方法设定

### 固定设定
- 上传对象：完整局部 Jacobian 或完整目标梯度矩阵
- 参与方式：部分客户端参与，保留全参与特例
- 本地更新：默认 `E=1`
- 服务器：先聚合 Jacobian，再计算全局下降方向
- 模型：小模型
- 目标数：优先 `m=2`，必要时 `m=3`

### 不做的事
- 不做压缩
- 不做本地多步漂移研究
- 不做复杂 benchmark
- 不做大模型

## 3. 核心实验问题

- Jacobian 维度是否正确
- 聚合后的 Jacobian 是否与理论定义一致
- 方向选择后是否真的能更新参数
- 目标曲线是否基本合理
- 不同 seed 下是否大致稳定

## 4. 数据集与任务

### 4.1 必做：合成二目标回归
用途：
- 最容易 debug
- 最容易画 Pareto 趋势
- 最容易定位数值错误

建议：
- 小 MLP 或线性模型
- `K=8,16`
- `m=2`
- `C=0.25,0.5,1.0`
- `E=1`

### 4.2 可选：小型多任务分类玩具
用途：验证分类场景下接口是否通顺。
若时间有限，可在阶段 1 跳过。

## 5. 必须采集的指标

- 每轮每个目标值
- Jacobian Frobenius norm
- 方向范数
- 是否出现 NaN/Inf
- 单轮运行时间
- 单轮上行/下行字节数

## 6. 基线对照

本阶段对照只需最低限度：
- 随机方向或简单平均梯度方向
- 现有实现中的占位聚合器版本

目的不是公平比较，而是确认 FedJD 确实不是空跑。

## 7. 成功判定

满足以下条件即视为阶段完成：
- 连续多轮训练无数值崩溃
- 至少一个玩具配置上主要目标有下降趋势
- Jacobian 聚合与参数更新链路可复现
- 指标日志可完整导出

## 8. 阶段交付物

- 运行脚本一份
- 合成任务结果图一组
- 维度/方向/日志检查清单一份
- 阶段总结 markdown 一份

## 9. 阶段风险

- 看似上传完整 Jacobian，实际上传的是随机 batch Jacobian
- 评估只使用局部 batch，导致曲线误判
- 当前方向选择器退化成 MGDA 风格，影响后续定位

## 10. 结束后必须回答的问题

- FedJD 是否稳定跑通
- 哪个模块最容易出 bug
- 当前实现中的“理论版本”和“实际运行版本”差距在哪里
